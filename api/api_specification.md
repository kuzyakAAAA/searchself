# Спецификация API системы бронирования

## 1. Создание бронирования
**URL:** `/api/v1/bookings`  
**Method:** `POST`  
**Description:** Создает новую запись о бронировании переговорной комнаты.

### Тело запроса (Request Body)
| Поле | Тип | Обяз. | Описание |
| :--- | :--- | :---: | :--- |
| `user_id` | Integer | Да | ID сотрудника, создающего бронь. |
| `room_id` | Integer | Да | ID переговорной комнаты. |
| `start_time` | String | Да | Начало в формате ISO 8601 (UTC). |
| `end_time` | String | Да | Окончание в формате ISO 8601 (UTC). |

**Пример запроса:**
```json
{
  "user_id": 1,
  "room_id": 42,
  "start_time": "2026-05-10T14:00:00Z",
  "end_time": "2026-05-10T15:00:00Z"
}
```

## 2. Ответы
### Успешное создание ###
```
{
  "id": 123,
  "status": "confirmed",
  "message": "Бронирование успешно создано"
}
```

### Ошибка: Конфликт времени ###
```
{
  "error_code": "BOOKING_CONFLICT",
  "message": "Выбранное время пересекается с существующим бронированием.",
  "details": {
    "conflicting_booking_id": 105,
    "current_occupant": "Иван Иванов"
  }
}
```
### Ошибка: Некорректный интервал ###
```
{
  "error_code": "INVALID_TIME_RANGE",
  "message": "Время окончания должно быть позже времени начала, и бронирование не может быть в прошлом."
}
```

### Ошибка: Ресурс не найден ###
```
{
  "error_code": "RESOURCE_NOT_FOUND",
  "message": "Указанный пользователь или комната не существуют в системе."
}
```

# Итог #
 - Спроектирован контракт взаимодействия (REST API): определен метод POST /api/v1/bookings для создания бронирования, включая структуру тела запроса в формате JSON.

 - Установлен стандарт передачи данных: зафиксировано использование формата ISO 8601 для меток времени и целочисленных идентификаторов для связи с базой данных.

 - Разработана система обработки исключений: сформированы специфические коды ошибок (BOOKING_CONFLICT, INVALID_TIME_RANGE) и структура ответов сервера для корректной обработки нарушений бизнес-правил на стороне интерфейса.

 - Сформирована матрица прав доступа (RBAC): разграничены полномочия ролей «Сотрудник» и «Администратор» для всех ключевых функций системы, включая правила отмены и модерации броней.

 - Создан единый технический регламент: вся документация сведена в структурированный файл api_and_rbac_specs.md, обеспечивающий прозрачность требований для разработки и тестирования.

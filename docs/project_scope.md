# Рамки проекта

## Название проекта

Система бронирования переговорных комнат.

## Проблема

В компании сотрудники используют переговорные комнаты для встреч, созвонов и командных обсуждений.

Если процесс бронирования не автоматизирован, могут возникать проблемы:

- несколько сотрудников могут претендовать на одну комнату в одно и то же время;
- сложно понять, какие комнаты свободны;
- нет единой истории бронирований;
- администратору сложно управлять списком комнат;
- сотрудники могут забывать об отмене ненужных бронирований.

## Цель системы

Создать систему, которая позволит сотрудникам удобно бронировать переговорные комнаты и снизит количество конфликтов при использовании комнат.

## Пользователи системы

### Сотрудник

Сотрудник может:

- просматривать список переговорных комнат;
- смотреть доступность комнаты;
- создавать бронирование;
- отменять свое бронирование;
- просматривать свои бронирования.

### Администратор

Администратор может:

- добавлять переговорные комнаты;
- редактировать информацию о комнатах;
- отключать комнаты;
- просматривать все бронирования.

### Система уведомлений

Система уведомлений может:

- отправлять подтверждение о создании бронирования;
- отправлять уведомление об отмене бронирования.

## Заинтересованные стороны

### Сотрудники компании

Используют систему для поиска свободных переговорных комнат и создания бронирований.

### Администратор офиса

Следит за актуальностью списка переговорных комнат, отключает недоступные комнаты и контролирует использование помещений.

### Руководители команд

Заинтересованы в том, чтобы сотрудники могли быстро находить свободные комнаты для встреч.

### Команда разработки

Использует аналитическую документацию для реализации системы: требований, бизнес-правил, модели данных и API.

### Команда тестирования

Использует требования, бизнес-правила и сценарии для подготовки тест-кейсов.

## Что входит в первую версию

В первую версию системы входит:

- просмотр списка переговорных комнат;
- просмотр информации о комнате;
- создание бронирования;
- проверка конфликта времени;
- отмена бронирования;
- управление комнатами администратором;
- хранение истории бронирований;
- базовое описание API;
- базовая модель данных.

## Что не входит в первую версию

В первую версию системы не входит:

- интеграция с корпоративным календарем;
- авторизация через SSO;
- мобильное приложение;
- сложная система уведомлений;
- повторяющиеся бронирования;
- бронирование оборудования;
- подтверждение бронирования руководителем;
- аналитические отчеты по загрузке комнат.

## Допущения

- В системе уже существует список сотрудников компании.
- В первой версии авторизация рассматривается упрощенно.
- Один пользователь может иметь несколько бронирований.
- Одна комната может иметь много бронирований в разные периоды времени.
- Бронирование создается только на конкретную дату и время.
- Повторяющиеся бронирования в первой версии не поддерживаются.
- Проверка доступности комнаты выполняется при создании бронирования.
- Уведомления рассматриваются как отдельный внешний или внутренний компонент.

## Основные ограничения

- Нельзя создать бронирование в прошлом.
- Нельзя создать бронирование, если комната уже занята в выбранный период.
- Пользователь может отменить только свое бронирование.
- Администратор может отключить комнату.
- Отключенную комнату нельзя забронировать.

## Критерии успеха проекта

Проект можно считать успешным, если:

- сотрудник может посмотреть список доступных переговорных комнат;
- сотрудник может создать бронирование свободной комнаты;
- система не позволяет создать бронирование при конфликте времени;
- сотрудник может отменить свое бронирование;
- администратор может управлять списком комнат;
- данные о бронированиях сохраняются в системе;
- разработчик может понять основную логику системы по документации;
- тестировщик может подготовить тест-кейсы на основе требований и бизнес-правил.

# Use Cases

## Назначение документа

Документ содержит описание основных сценариев использования системы бронирования переговорных комнат.

Use cases помогают понять, какие действия разные участники могут выполнять в системе.

---

## Акторы

### Сотрудник

Пользователь системы, который может просматривать переговорные комнаты, создавать бронирования и отменять свои бронирования.

### Администратор

Пользователь системы, который управляет списком переговорных комнат и может просматривать бронирования.

### Система уведомлений

Внешний или внутренний компонент, который получает события о создании и отмене бронирований.

---

## UC-001. Просмотр списка переговорных комнат

### Актор

Сотрудник

### Описание

Сотрудник открывает список переговорных комнат, чтобы выбрать подходящую комнату для встречи.

### Предусловия

- Система доступна.
- В системе есть список переговорных комнат.

### Основной сценарий

1. Сотрудник открывает раздел со списком переговорных комнат.
2. Система запрашивает список комнат.
3. Система отображает список комнат с основной информацией.
4. Сотрудник просматривает список.

### Результат

Сотрудник видит список переговорных комнат.

### Связанные требования

- FR-001. Просмотр списка переговорных комнат

## UC-003. Проверка доступности комнаты

### Актор

Сотрудник

### Описание

Сотрудник проверяет, свободна ли выбранная переговорная комната в нужный период времени.

### Предусловия

- Комната существует в системе.
- Сотрудник указал дату, время начала и время окончания.

### Основной сценарий

1. Сотрудник выбирает комнату.
2. Сотрудник указывает дату и время.
3. Система проверяет корректность временного интервала.
4. Система проверяет существующие бронирования.
5. Система показывает результат проверки доступности.

### Альтернативный сценарий

Если время указано некорректно, система показывает сообщение об ошибке.

### Результат

Сотрудник понимает, доступна ли комната в выбранный период.

### Связанные требования и правила

- FR-003. Проверка доступности комнаты
- BR-002. Время окончания должно быть позже времени начала
- BR-003. Нельзя создать пересекающееся бронирование

## UC-004. Создание бронирования

### Актор

Сотрудник

### Описание

Сотрудник создает бронирование переговорной комнаты на выбранные дату и время.

### Предусловия

- Сотрудник выбрал комнату.
- Сотрудник указал дату, время начала и время окончания.
- Комната активна.
- Комната свободна в выбранный период.

### Основной сценарий

1. Сотрудник выбирает комнату.
2. Сотрудник указывает дату и время бронирования.
3. Система проверяет, что время начала не находится в прошлом.
4. Система проверяет, что время окончания позже времени начала.
5. Система проверяет, что комната активна.
6. Система проверяет, что комната свободна.
7. Система создает бронирование.
8. Система сохраняет бронирование.
9. Система передает событие `booking_created` в систему уведомлений.
10. Система показывает сотруднику подтверждение.

### Альтернативные сценарии

- Если время начала находится в прошлом, система показывает ошибку.
- Если время окончания меньше или равно времени начала, система показывает ошибку.
- Если комната отключена, система показывает ошибку.
- Если комната уже занята, система показывает ошибку.

### Результат

Бронирование создано или сотрудник получил сообщение об ошибке.

### Связанные требования и правила

- FR-004. Создание бронирования
- FR-010. Отправка уведомления о бронировании
- BR-001. Нельзя создать бронирование в прошлом
- BR-002. Время окончания должно быть позже времени начала
- BR-003. Нельзя создать пересекающееся бронирование
- BR-004. Отключенную комнату нельзя забронировать
- BR-010. После создания бронирования система должна передать событие в систему уведомлений

## UC-005. Просмотр своих бронирований

### Актор

Сотрудник

### Описание

Сотрудник просматривает список своих бронирований.

### Предусловия

### Основной сценарий

1. Сотрудник открывает раздел своих бронирований.
2. Система запрашивает бронирования сотрудника.
3. Система отображает список бронирований.
4. Сотрудник просматривает активные, завершенные и отмененные бронирования.

### Результат

Сотрудник видит свои бронирования.

### Связанные требования и правила

- FR-005. Просмотр своих бронирований
- BR-012. История бронирований должна сохраняться

## UC-006. Отмена бронирования

### Актор

Сотрудник

### Описание

Сотрудник отменяет свое бронирование, если встреча больше не нужна.

### Предусловия

- Бронирование существует.
- Бронирование принадлежит сотруднику.
- Бронирование не отменено.
- Бронирование еще не завершено.

### Основной сценарий

1. Сотрудник открывает список своих бронирований.
2. Сотрудник выбирает бронирование.
3. Сотрудник нажимает отмену бронирования.
4. Система проверяет, что бронирование принадлежит сотруднику.
5. Система проверяет, что бронирование еще можно отменить.
6. Система меняет статус бронирования на `cancelled`.
7. Система передает событие `booking_cancelled` в систему уведомлений.
8. Система показывает подтверждение отмены.

### Альтернативные сценарии

- Если сотрудник пытается отменить чужое бронирование, система показывает ошибку.
- Если бронирование уже отменено, система показывает ошибку.
- Если бронирование уже завершено, система показывает ошибку.

### Результат

Бронирование отменено или сотрудник получил сообщение об ошибке.

### Связанные требования и правила

- FR-006. Отмена бронирования
- FR-010. Отправка уведомления о бронировании
- BR-005. Пользователь может отменить только свое бронирование
- BR-006. Нельзя отменить уже отмененное бронирование
- BR-007. Нельзя отменить завершенное бронирование
- BR-011. После отмены бронирования система должна передать событие в систему уведомлений
- BR-012. История бронирований должна сохраняться

## UC-007. Добавление переговорной комнаты

### Актор

Администратор

### Описание

Администратор добавляет новую переговорную комнату в систему.

### Предусловия

- Пользователь имеет роль администратора.

### Основной сценарий

1. Администратор открывает раздел управления комнатами.
2. Администратор нажимает добавление новой комнаты.
3. Администратор указывает название, вместимость, расположение и описание.
4. Система проверяет корректность данных.
5. Система создает новую комнату со статусом `active`.
6. Система отображает новую комнату в списке.

### Альтернативный сценарий

Если вместимость меньше или равна нулю, система показывает ошибку.

### Результат

Новая переговорная комната добавлена в систему.

### Связанные требования и правила

- FR-007. Добавление переговорной комнаты
- BR-008. Администратор может управлять комнатами
- BR-009. Вместимость комнаты должна быть больше нуля

## UC-009. Отключение переговорной комнаты

### Актор

Администратор

### Описание

Администратор отключает переговорную комнату, чтобы сотрудники не могли создавать для нее новые бронирования.

### Предусловия

- Пользователь имеет роль администратора.
- Комната существует в системе.

### Основной сценарий

1. Администратор открывает список комнат.
2. Администратор выбирает комнату.
3. Администратор меняет статус комнаты на `inactive`.
4. Система сохраняет новый статус комнаты.
5. Система скрывает или блокирует комнату для новых бронирований.

### Результат

Комната отключена для новых бронирований.

### Связанные требования и правила

- FR-009. Отключение переговорной комнаты
- BR-004. Отключенную комнату нельзя забронировать
- BR-008. Администратор может управлять комнатами

# Пользовательские истории

## Назначение документа

Документ содержит пользовательские истории для системы бронирования переговорных комнат.

Пользовательская история описывает потребность пользователя в формате:

Как <роль>, я хочу <действие>, чтобы <ценность>.

Пользовательские истории помогают понять, какие задачи пользователи хотят решать с помощью системы.

---

## US-001. Просмотр списка переговорных комнат

Как сотрудник, я хочу просматривать список переговорных комнат, чтобы выбрать подходящую комнату для встречи.

### Критерии приемки

- Пользователь видит список переговорных комнат.
- Для каждой комнаты отображается название, вместимость и расположение.
- Отключенные комнаты не должны быть доступны для создания нового бронирования.

### Связанные требования

- FR-001. Просмотр списка переговорных комнат

---

## US-002. Просмотр информации о комнате

Как сотрудник, я хочу просматривать подробную информацию о переговорной комнате, чтобы понять, подходит ли она для моей встречи.

### Критерии приемки

- Пользователь может открыть карточку выбранной комнаты.
- В карточке отображается название, вместимость, расположение, описание и статус активности.
- Пользователь понимает, можно ли использовать комнату для бронирования.

### Связанные требования

- FR-002. Просмотр информации о конкретной комнате

---

## US-003. Проверка доступности комнаты

Как сотрудник, я хочу проверить доступность комнаты на выбранное время, чтобы понять, могу ли я ее забронировать.

### Критерии приемки

- Пользователь указывает комнату, дату, время начала и время окончания.
- Система проверяет, есть ли конфликт с существующими бронированиями.
- Если комната свободна, система показывает, что бронирование возможно.
- Если комната занята, система показывает, что выбранное время недоступно.

### Связанные требования

- FR-003. Проверка доступности комнаты
- BR-003. Нельзя создать пересекающееся бронирование

---

## US-004. Создание бронирования

Как сотрудник, я хочу забронировать переговорную комнату, чтобы провести встречу в выбранное время.

### Критерии приемки

- Пользователь указывает комнату, дату, время начала и время окончания.
- Система проверяет корректность введенного времени.
- Система проверяет, что комната активна.
- Система проверяет, что комната свободна в выбранный период.
- Если все условия выполнены, система создает бронирование.
- Если есть ошибка, система показывает понятное сообщение.

### Связанные требования

- FR-004. Создание бронирования
- BR-001. Нельзя создать бронирование в прошлом
- BR-002. Время окончания должно быть позже времени начала
- BR-003. Нельзя создать пересекающееся бронирование
- BR-004. Отключенную комнату нельзя забронировать

---

## US-005. Просмотр своих бронирований

Как сотрудник, я хочу просматривать свои бронирования, чтобы видеть запланированные встречи и историю использования комнат.

### Критерии приемки

- Пользователь видит список своих бронирований.
- Для каждого бронирования отображается комната, дата, время начала, время окончания и статус.
- Пользователь видит активные, отмененные и завершенные бронирования.

### Связанные требования

- FR-005. Просмотр своих бронирований
- BR-012. История бронирований должна сохраняться

---

## US-006. Отмена бронирования

Как сотрудник, я хочу отменить свое бронирование, чтобы освободить комнату, если встреча больше не нужна.

### Критерии приемки

- Пользователь может отменить только свое бронирование.
- Пользователь не может отменить чужое бронирование.
- Пользователь не может повторно отменить уже отмененное бронирование.
- Пользователь не может отменить завершенное бронирование.
- После успешной отмены статус бронирования меняется на "cancelled".

### Связанные требования

- FR-006. Отмена бронирования
- BR-005. Пользователь может отменить только свое бронирование
- BR-006. Нельзя отменить уже отмененное бронирование
- BR-007. Нельзя отменить завершенное бронирование
- BR-012. История бронирований должна сохраняться

---

## US-007. Добавление переговорной комнаты

Как администратор, я хочу добавлять новые переговорные комнаты, чтобы сотрудники могли бронировать новые доступные помещения.

### Критерии приемки

- Администратор может указать название комнаты, вместимость, расположение и описание.
- Вместимость комнаты должна быть больше нуля.
- После создания комната получает активный статус.
- Новая комната появляется в списке переговорных комнат.

### Связанные требования

- FR-007. Добавление переговорной комнаты
- BR-008. Администратор может управлять комнатами
- BR-009. Вместимость комнаты должна быть больше нуля

---

## US-008. Редактирование переговорной комнаты

Как администратор, я хочу редактировать информацию о переговорной комнате, чтобы поддерживать данные о комнатах в актуальном состоянии.

### Критерии приемки

- Администратор может изменить название, вместимость, расположение, описание и статус активности комнаты.
- Вместимость комнаты должна быть больше нуля.
- После сохранения изменений обновленная информация отображается в системе.

### Связанные требования

- FR-008. Редактирование переговорной комнаты
- BR-008. Администратор может управлять комнатами
- BR-009. Вместимость комнаты должна быть больше нуля

---

## US-009. Отключение переговорной комнаты

Как администратор, я хочу отключать переговорную комнату, чтобы сотрудники не могли бронировать помещение, которое временно недоступно.

### Критерии приемки

- Администратор может изменить статус комнаты на "inactive".
- Отключенная комната не должна быть доступна для новых бронирований.
- Уже существующие бронирования не удаляются из истории.

### Связанные требования

- FR-009. Отключение переговорной комнаты
- BR-004. Отключенную комнату нельзя забронировать
- BR-008. Администратор может управлять комнатами

---

## US-010. Получение уведомления о создании бронирования

Как сотрудник, я хочу получить уведомление после создания бронирования, чтобы убедиться, что комната успешно закреплена за мной.

### Критерии приемки

- После успешного создания бронирования система формирует событие booking_created.
- Событие передается в систему уведомлений.
- Уведомление содержит информацию о комнате, дате и времени бронирования.

### Связанные требования

- FR-010. Отправка уведомления о бронировании
- BR-010. После создания бронирования система должна передать событие в систему уведомлений

---

## US-011. Получение уведомления об отмене бронирования

Как сотрудник, я хочу получить уведомление после отмены бронирования, чтобы убедиться, что комната больше не закреплена за мной.

### Критерии приемки

- После успешной отмены бронирования система формирует событие booking_cancelled.
- Событие передается в систему уведомлений.
- Уведомление содержит информацию о комнате, дате и времени отмененного бронирования.

### Связанные требования

- FR-010. Отправка уведомления о бронировании
- BR-011. После отмены бронирования система должна передать событие в систему уведомлений

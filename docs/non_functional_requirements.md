# Нефункциональные требования

## Назначение документа

Документ содержит нефункциональные требования к системе бронирования переговорных комнат.

Нефункциональные требования описывают не конкретные функции системы, а качество ее работы: производительность, надежность, безопасность, удобство, хранение данных и другие характеристики.

---

## NFR-001. Производительность

Система должна обрабатывать основные пользовательские запросы не дольше 2 секунд при нормальной нагрузке.

### Примеры запросов

- просмотр списка переговорных комнат;
- просмотр информации о комнате;
- создание бронирования;
- отмена бронирования;
- просмотр своих бронирований.

### Обоснование

Пользователь должен быстро получать ответ от системы, чтобы процесс бронирования не занимал много времени.

---

## NFR-002. Доступность системы

Система должна быть доступна пользователям в рабочее время компании.

### Рабочее время

С понедельника по пятницу с 09:00 до 19:00.

### Обоснование

Сотрудники используют переговорные комнаты в течение рабочего дня, поэтому система должна быть доступна в это время.

---

## NFR-003. Надежность хранения данных

Система должна сохранять данные о созданных, отмененных и завершенных бронированиях.

### Какие данные должны храниться

- пользователь, создавший бронирование;
- переговорная комната;
- время начала бронирования;
- время окончания бронирования;
- статус бронирования;
- дата и время создания бронирования.

### Обоснование

История бронирований нужна для просмотра, контроля и возможного анализа использования переговорных комнат.

---

## NFR-004. Безопасность доступа

Пользователь должен иметь доступ только к тем действиям, которые разрешены его ролью.

### Примеры ограничений

- сотрудник может отменить только свое бронирование;
- сотрудник не может добавлять или отключать переговорные комнаты;
- администратор может управлять переговорными комнатами;
- администратор может просматривать все бронирования.

### Обоснование

Система должна защищать данные и действия от несанкционированного доступа.

---

## NFR-005. Удобство использования

Интерфейс системы должен быть понятным для сотрудника без дополнительного обучения.

### Основные требования к удобству

- пользователь должен быстро находить список комнат;
- пользователь должен видеть статус доступности комнаты;
- пользователь должен получать понятное сообщение при ошибке;
- пользователь должен понимать, успешно ли создано бронирование.

### Обоснование

Система предназначена для повседневного использования сотрудниками компании, поэтому процесс бронирования должен быть простым.

---

## NFR-006. Понятные сообщения об ошибках

Система должна возвращать понятные сообщения об ошибках при невозможности выполнить действие.

### Примеры ошибок

- комната уже занята в выбранный период;
- комната отключена;
- время окончания раньше времени начала;
- пользователь пытается создать бронирование в прошлом;
- пользователь пытается отменить чужое бронирование.

### Обоснование

Понятные ошибки помогают пользователю понять, почему действие не выполнено, и что нужно исправить.

---

## NFR-007. Логирование

Система должна фиксировать основные события и ошибки в логах.

### Что нужно логировать

- создание бронирования;
- отмену бронирования;
- попытку создать бронирование при конфликте времени;
- ошибки при работе системы.

### Обоснование

Логи нужны для диагностики проблем, анализа ошибок и поддержки системы.

---

## NFR-008. Масштабируемость

Система должна допускать добавление новых переговорных комнат и пользователей без изменения основной логики работы.

### Обоснование

Количество сотрудников и комнат может увеличиваться, поэтому система должна быть готова к расширению.

---

## NFR-009. Совместимость с API

Система должна предоставлять REST API для взаимодействия с клиентским интерфейсом и другими компонентами.

### Обоснование

REST API позволит frontend-приложению, системе уведомлений и другим компонентам взаимодействовать с backend-частью системы.

---

## NFR-010. Поддерживаемость

Документация, модель данных и API должны быть описаны так, чтобы разработчики и тестировщики могли использовать их для реализации и проверки системы.

### Обоснование

Проект должен быть понятен не только автору, но и другим участникам команды.
